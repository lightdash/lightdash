## Cypress command expansion

- `cy.login()` (`packages/e2e/cypress/cli/api/commands.cy.ts:22`) is defined at `packages/e2e/cypress/support/commands.ts:152-172`. It creates/restores a `cy.session` keyed by the seed admin email, POSTs `api/v1/login` with `SEED_ORG_1_ADMIN_EMAIL` and `SEED_ORG_1_ADMIN_PASSWORD`, requires status 200, and validates restored sessions with GET `api/v1/user` status 200. The seed is the registered org-1 admin, UUID `b264d83a-9000-426a-85ec-3f9c20f368ce`, email `demo@lightdash.com`, with admin role (`packages/common/src/index.ts:465-481`).
- `cy.getApiToken()` (`packages/e2e/cypress/cli/api/commands.cy.ts:23`) is defined at `packages/e2e/cypress/support/commands.ts:375-390`. It POSTs `api/v1/user/me/personal-access-tokens` with `{ description: 'e2e', autoGenerated: true, expiresAt: null }`, requires status 200, and yields only `results.token`; it discards the PAT UUID needed for deletion.
- PAT creation requires a permitted registered account (`packages/backend/src/services/PersonalAccessTokenService.ts:58-89`). The controller explicitly disallows PAT-authenticated PAT creation, while the Cypress cookie session is accepted (`packages/backend/src/controllers/userController.ts:648-675`).
- `cy.exec` is Cypress built-in, not a repository custom command. All three invocations are shell commands (`packages/e2e/cypress/cli/api/commands.cy.ts:6,14,24`). Login passes `failOnNonZeroExit: false` and subprocess-only environment overrides `NODE_ENV=development` and `CI=true` (`packages/e2e/cypress/cli/api/commands.cy.ts:24-30`).
- No aliases, fixtures, hooks, intercepts, or shared variables other than module constants are used. Cypress chaining enforces login -> PAT creation -> subprocess -> stderr assertion for the login case (`packages/e2e/cypress/cli/api/commands.cy.ts:22-32`).

## State, seed, and environment assumptions

- `lightdash` must resolve on `PATH`; the package exposes `dist/index.js` as its binary (`packages/cli/package.json:10-12`). CI explicitly builds and globally installs the CLI before this spec (`.github/workflows/pr.yml:662-667`). Local execution has the same undocumented global-install/PATH requirement.
- The module reads `Cypress.config('baseUrl')` once (`packages/e2e/cypress/cli/api/commands.cy.ts:2`). It is `http://localhost:3000` by default (`packages/e2e/cypress.config.ts:23-26`) and the CI lane overrides it with the PR preview URL (`.github/workflows/pr.yml:673-679`).
- The login test requires the seeded org-1 admin and a valid password/session (`packages/e2e/cypress/support/commands.ts:152-170`; `packages/common/src/index.ts:465-481`). The user must remain registered and able to create its own PAT (`packages/backend/src/services/PersonalAccessTokenService.ts:63-89`).
- PAT creation is a persistent DB insert. Tokens are random and descriptions are not used as identifiers, so duplicate description `e2e` does not collide, but every run/retry accumulates another token because `expiresAt` is null and no delete runs (`packages/e2e/cypress/support/commands.ts:375-389`; `packages/backend/src/models/DashboardModel/PersonalAccessTokenModel.ts:109-147`).
- The CLI validates the token with GET `/api/v1/user` (`packages/cli/src/handlers/login.ts:76-97`), then stores the token and URL and stores the returned user/org (`packages/cli/src/handlers/login.ts:287-310`). Config is a plaintext YAML file at `~/.config/lightdash/config.yaml`, rewritten rather than transactionally updated (`packages/cli/src/config.ts:8-9,36-39,135-155`). There is no isolated `HOME` or config cleanup in the source spec.
- `CI=true` is behaviorally significant, not just metadata: it makes the CLI noninteractive (`packages/cli/src/globalState.ts:32-38`) and causes login to select the first project (`packages/cli/src/handlers/login.ts:312-318`). `setFirstProject` assumes `/api/v1/org/projects` returns at least one item and persists index 0 (`packages/cli/src/handlers/setProject.ts:101-119`). No project UUID/name is asserted, and ordering changes can alter local config.
- `NODE_ENV=development` only selects the development analytics write key; analytics still makes external requests to `https://analytics.lightdash.com` and swallows errors (`packages/cli/src/analytics/analytics.ts:571-610,613-635`). Cypress `blockHosts` applies to browser traffic, not the CLI subprocess (`packages/e2e/cypress.config.ts:27-34`).
- The three tests are declared in help/version/login order and have no hooks or explicit cross-test data dependency (`packages/e2e/cypress/cli/api/commands.cy.ts:4-34`). Login can still be affected by config left by earlier suites or concurrent processes because the CLI uses the shared OS home. It does not require either earlier test to pass.
- Run-mode retries default to two (`packages/e2e/cypress.config.ts:12-21`). A login retry creates an additional PAT. `cy.session` may reuse the cookie session, but `cy.getApiToken()` is outside the session callback and always mutates.

## Synchronization and timeout requirements

- There are no sleeps, intercept waits, debounce waits, polling loops, or custom per-command timeouts. Cypress serializes each chain.
- The config sets `defaultCommandTimeout` to 10 seconds but does not set `execTimeout` (`packages/e2e/cypress.config.ts:15-26`). The port should give subprocesses an explicit bounded timeout and include captured stdout/stderr in failures.
- Login waits on several sequential network operations: Cypress login, PAT creation, CLI token validation, analytics, project listing, and config writes (`packages/e2e/cypress/cli/api/commands.cy.ts:22-32`; `packages/cli/src/handlers/login.ts:227-233,249-318`). External analytics errors are swallowed but requests are awaited, so egress can delay the command.
- The login assertion is emitted before project selection (`packages/cli/src/handlers/login.ts:310-318`), and project-selection exceptions are swallowed (`packages/cli/src/handlers/login.ts:337-342`). The replacement must decide whether successful authentication alone or complete project selection is the contract.

## Locator and strictness risks

There are no DOM locators, accessibility selectors, frames, canvases, SVGs, Monaco editors, or virtualized lists.

Process-output strictness risks remain:

- Help checks a prose substring (`packages/e2e/cypress/cli/api/commands.cy.ts:6-8`), so copy changes fail even when the command is healthy.
- Version only checks `0.` (`packages/e2e/cypress/cli/api/commands.cy.ts:14-16`), which can accept the wrong 0.x release. CI's exact comparison is stronger (`.github/workflows/pr.yml:668-672`).
- Login disables nonzero-exit failure and checks only stderr (`packages/e2e/cypress/cli/api/commands.cy.ts:24-32`). A process that prints the success line and later exits nonzero can pass. The Node replacement should require exit code 0 first, then assert the success marker and persisted server URL.
- URL and token are interpolated into one shell string without quoting (`packages/e2e/cypress/cli/api/commands.cy.ts:24`). Current generated PATs are constrained random values, but the replacement should use `spawn`/`execFile` argument arrays rather than shell interpolation.

## Nonstandard or surprising behavior

- These are browser-runner tests that never start from browser behavior; Cypress is only a shell/API harness.
- The login success message intentionally goes to stderr (`packages/cli/src/handlers/login.ts:310`), which is why the test does not inspect stdout.
- `failOnNonZeroExit: false` makes the test materially weaker than its title suggests (`packages/e2e/cypress/cli/api/commands.cy.ts:24-32`).
- Login performs a best-effort version health check before token validation (`packages/cli/src/handlers/login.ts:186-191`; `packages/cli/src/handlers/dbt/apiClient.ts:331-357`). That check uses any preexisting CLI context and suppresses failures, another reason to isolate `HOME`.
- The PAT request marks the token `autoGenerated: true`, but persistence stores description/expiry/hash and does not deduplicate autogenerated tokens (`packages/backend/src/models/DashboardModel/PersonalAccessTokenModel.ts:126-147`).
- No downloads/uploads, popups, iframes, clipboard, drag-and-drop, browser APIs, timezone behavior, or UI rendering are involved.

## Exact port plan

1. Add `packages/cli/scripts/smoke-api-commands.mjs` as one self-contained Node script; do not add Playwright files or shared helpers.
2. Invoke the built artifact with `process.execPath` plus `packages/cli/dist/index.js`, avoiding global `PATH` and shell interpolation.
3. Run `help`, require exit code 0, and assert the description from `packages/cli/src/index.ts:104-109`.
4. Do not duplicate the version case in the script unless local parity is required; `.github/workflows/pr.yml:668-672` already compares the exact package and executable versions. If retained, read `packages/cli/package.json` and compare exact trimmed output.
5. For login, create a temporary directory and pass it as `HOME`; use `SITE_URL` (default `http://localhost:3000`) rather than a Playwright base URL.
6. In the script, POST `/api/v1/login` as the seed admin, retain the session cookie, POST `/api/v1/user/me/personal-access-tokens`, and retain both token and UUID. Keep this setup local rather than introducing an API helper used once.
7. Spawn `login <SITE_URL> --token <token>` with `CI=true` and `NODE_ENV=development`; require exit code 0, stderr `Login successful`, and config YAML containing the normalized server URL. Optionally assert selected project only after the open question below is resolved.
8. In `finally`, DELETE `/api/v1/user/me/personal-access-tokens/{uuid}` with the cookie session and recursively remove the temporary home, including on assertion/subprocess failure.
9. Update `.github/workflows/pr.yml` in `cli-tests-without-dbt` to run the Node smoke script after CLI build and exact version validation, passing the same preview URL currently supplied as Cypress `baseUrl` (`.github/workflows/pr.yml:666-679`). Keep the remaining Cypress CLI specs untouched.
10. Once the replacement is green, remove only `packages/e2e/cypress/cli/api/commands.cy.ts`. No Playwright config/auth/storage-state changes are needed.
