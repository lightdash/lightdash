## Cypress command expansion

### `cy.login()`

- Implemented at `packages/e2e/cypress/support/commands.ts:152-172`.
- Uses `cy.session` keyed by `SEED_ORG_1_ADMIN_EMAIL.email`, POSTs `api/v1/login` with the seeded admin email/password, requires HTTP 200, and validates reused sessions with GET `api/v1/user` returning 200 (`packages/e2e/cypress/support/commands.ts:153-169`).
- The credentials are `demo@lightdash.com` / `demo_password!` (`packages/common/src/index.ts:474-479`).
- No browser navigation occurs; authentication is a cookie-backed API session.

### `cy.getApiToken()`

- Implemented at `packages/e2e/cypress/support/commands.ts:375-389`.
- POSTs `api/v1/user/me/personal-access-tokens` with `{ description: 'e2e', autoGenerated: true, expiresAt: null }`, requires HTTP 200, and yields only `resp.body.results.token` (`packages/e2e/cypress/support/commands.ts:376-388`).
- Because only the token string is returned, the PAT UUID needed by DELETE `/api/v1/user/me/personal-access-tokens/{personalAccessTokenUuid}` is discarded. The backend exposes that DELETE at `packages/backend/src/controllers/userController.ts:680-697`.
- PAT creation does not accept PAT authentication and is guarded against demo mode (`packages/backend/src/controllers/userController.ts:648-657`), so the cookie session from `cy.login()` is material.

All other Cypress calls are built-ins: `cy.exec`, `cy.request`, and chained assertions. There are no aliases.

## State, seed, and environment assumptions

- **Working directory/PATH:** `projectDir` is `../../examples/snowflake-template` (`packages/e2e/cypress/cli/yaml-only/cli.cy.ts:8-9`), relying on execution from `packages/e2e`. The bare `lightdash` command must resolve on `PATH`. CI explicitly builds and globally installs the CLI before running this lane (`.github/workflows/pr.yml:666-679`).
- **Fixture:** the project config declares Snowflake (`examples/snowflake-template/lightdash.config.yml:1-2`), while the only model is `users`, backed by the literal relation `DB.SCHEMA.USERS` with one metric and two dimensions (`examples/snowflake-template/lightdash/models/users.yml:3-26`). No actual Snowflake credentials or dbt installation are required because YAML-only mode sets `warehouseCredentials: false`, `skipDbtCompile: true`, and `skipWarehouseCatalog: true` (`packages/cli/src/lightdash/projectType.ts:108-119`).
- **Backend/seed:** deploy requires a reachable Lightdash instance at Cypress `baseUrl`; repository default is `http://localhost:3000` (`packages/e2e/cypress.config.ts:26`). It assumes seeded org 1 and its active admin account. Admin must be able to create/delete projects and create PATs; the instance cannot be in the backend's demo-restricted mode.
- **Environment:** both commands set `CI=true` and `NODE_ENV=development`; deploy additionally sets `LIGHTDASH_API_KEY` and `LIGHTDASH_URL` (`packages/e2e/cypress/cli/yaml-only/cli.cy.ts:19-20,54-57`). `CI === 'true'` makes the CLI non-interactive (`packages/cli/src/globalState.ts:36-37`). `LIGHTDASH_API_KEY` and `LIGHTDASH_URL` override persisted config (`packages/cli/src/config.ts:75-79`).
- **Server mutations:** project creation POSTs `/api/v1/org/projects` (`packages/cli/src/handlers/createProject.ts:396-419`); deploy PUTs tags and parameters and then explores (`packages/cli/src/handlers/deploy.ts:101-128,428-437`). Missing defaults/table groups mean their conditional requests are skipped (`packages/cli/src/handlers/deploy.ts:133-158`). Cleanup DELETEs `/api/v1/org/projects/{projectUuid}`, whose route requires authentication and deletes by UUID (`packages/backend/src/routers/organizationRouter.ts:58-73`).
- **Local mutation:** config is fixed at `~/.config/lightdash/config.yaml` (`packages/cli/src/config.ts:8-9`). Deploy calls `setProject` after creation (`packages/cli/src/handlers/deploy.ts:696-713`), so concurrent CLI tests sharing a home directory can overwrite each other's selected project/config.
- **Duplicate-name risk:** every current run creates `YAML-only e2e test` (`packages/e2e/cypress/cli/yaml-only/cli.cy.ts:50-51`). Creation and cleanup use returned UUIDs rather than name lookup, but duplicate names make leaked records hard to distinguish. The port should generate a run-unique name.
- **External service:** CLI analytics POSTs to `https://analytics.lightdash.com/v1/track` even in CI and only swallows failures after awaiting fetch (`packages/cli/src/analytics/analytics.ts:578-609`). `NODE_ENV=development` selects the development analytics key (`packages/cli/src/analytics/analytics.ts:571-576`). This is best-effort but can add network latency; no warehouse service is contacted.

## Synchronization and timeout requirements

- Cypress config sets a 10-second default command timeout and two run-mode retries by default (`packages/e2e/cypress.config.ts:13,18-22`), but the two `cy.exec` calls provide no explicit timeout (`packages/e2e/cypress/cli/yaml-only/cli.cy.ts:16-21,49-59`).
- Playwright's default test timeout is not overridden, while its config only sets assertion/action/navigation timeouts (`packages/e2e/playwright.config.ts:11,21-23`). The CLI deploy includes health checking, compilation, project creation, multiple API writes, and analytics, so the target should set an explicit per-test timeout (at least 60 seconds, with final value confirmed from CI timings) and give the spawned process an abort/kill timeout.
- Assertions must happen only after process close so all stderr is collected. Capture stdout and stderr separately because current behavior specifically asserts stderr.
- Cleanup must be registered before risky assertions and run in `finally`. Currently assertions precede UUID parsing (`packages/e2e/cypress/cli/yaml-only/cli.cy.ts:62-70`), so an assertion failure can leak the project.
- Cypress retries can create a PAT and project per attempt, while the suite closure stores at most one project UUID and no PAT UUID (`packages/e2e/cypress/cli/yaml-only/cli.cy.ts:34,67-71`). The port must make each retry independently clean its own resources.
- No debounce, network intercept/wait alias, polling loop, custom wait, or browser rendering synchronization exists.

## Locator and strictness risks

- There are no DOM locators, so Playwright locator strictness is not applicable.
- `stderr` containing the generic word `users` is weak evidence: it does not prove the compiled explore is specifically the fixture model (`packages/e2e/cypress/cli/yaml-only/cli.cy.ts:24,62`). Preserve parity initially, but prefer a more specific stable CLI output assertion only if the CLI contract exposes one.
- `Successfully deployed` is a substring of current human-readable output, not a structured contract (`packages/e2e/cypress/cli/yaml-only/cli.cy.ts:63`; `packages/cli/src/handlers/deploy.ts:728-740`). ANSI styling currently still permits substring matching but output wording/style changes can break the test.
- `/projectUuid=([\w-]*)/` accepts an empty value and non-UUID word characters, and parsing failure does not fail the test (`packages/e2e/cypress/cli/yaml-only/cli.cy.ts:67-71`). The port should require a canonical UUID match and fail explicitly if it is absent.
- The current cleanup request has no status/body assertion (`packages/e2e/cypress/cli/yaml-only/cli.cy.ts:36-43`). The port should require cleanup responses to be OK while retaining the original test failure as the primary diagnostic.

## Nonstandard or surprising behavior

- This E2E spec never renders a UI. Cypress is only a process/API orchestrator.
- `failOnNonZeroExit: false` is intentional so the test can inspect stderr and exit code itself (`packages/e2e/cypress/cli/yaml-only/cli.cy.ts:17,52`). A Node port must preserve the result on nonzero exit rather than letting `execFile` rejection bypass assertions.
- Successful CLI messages are written to stderr (`packages/cli/src/handlers/compile.ts:545-551`; `packages/cli/src/handlers/deploy.ts:738-740`). Do not silently switch assertions to stdout.
- The deploy command both mutates the server and persists the created project as the user's active local CLI project (`packages/cli/src/handlers/deploy.ts:696-713`). Cleanup removes the server project but does not restore CLI config.
- The compile path is not entirely side-effect-free: completed analytics identifies through `getConfig`, which can create the local anonymous UUID/config file (`packages/cli/src/analytics/analytics.ts:28-29`; `packages/cli/src/config.ts:67-69`).
- The success URL is used as an implicit machine-readable cleanup channel even though it is presentation output (`packages/cli/src/handlers/deploy.ts:724-740`).
- No downloads/uploads, popups, iframes, clipboard, canvas/SVG, Monaco, virtualization, drag-and-drop, browser APIs, timezone behavior, or UI authentication state are involved.

1. `packages/e2e/playwright.config.ts` currently has only a `firefox` project matching every `**/*.spec.ts`, depending on browser auth setup and stored admin state (`packages/e2e/playwright.config.ts:28-40`). No separate database/preview is required if each target test uses a unique project name, tracks returned UUIDs, cleans project and PAT in `finally`, and isolates local CLI config. Server-side operations are scoped to the created UUID.

## Exact port plan

1. After the `playwright-cli-node-lane` coordinator lands its config/CI prerequisite, create only `packages/e2e/playwright/cli/yaml-only/cli.spec.ts` for the port.
2. Add a file-local Node process runner using argument arrays (no shell interpolation) that:
   - runs `lightdash` from `packages/e2e` so current project-dir semantics remain valid;
   - captures stdout, stderr, and numeric exit code even on failure;
   - has an explicit abort timeout and terminates the child on timeout;
   - passes string environment values and a per-test temporary `HOME` while preserving required parent `PATH`.
3. Port compile one-for-one: execute `compile --project-dir ../../examples/snowflake-template` with `CI=true` and `NODE_ENV=development`; assert exit code 0 and the two current stderr markers.
4. Keep deploy self-contained rather than depending on suite order or Playwright's browser storage state:
   - POST `/api/v1/login` with the seeded admin credentials and require OK;
   - POST `/api/v1/user/me/personal-access-tokens`, retaining both PAT UUID and secret token;
   - use `PLAYWRIGHT_BASE_URL ?? http://localhost:3000`, matching config fallback (`packages/e2e/playwright.config.ts:21`);
   - generate a unique project name and run `deploy --create <name> --project-dir ../../examples/snowflake-template` with `CI`, `NODE_ENV`, `LIGHTDASH_API_KEY`, and `LIGHTDASH_URL`;
   - assert exit code/output, require a canonical project UUID from stderr, and retain it immediately.
5. In `finally`, DELETE the created project by UUID if creation reached that point, DELETE the PAT by its UUID, assert both cleanup responses, and remove the temporary home directory. Cleanup must execute for failed assertions, process failures, timeouts, and Playwright retries.
6. Do not introduce browser/page fixtures, selectors, shared UI auth helpers, name-based deletion, arbitrary sleeps, or an API-tests replacement: the behavior under test is CLI compilation/deployment.
7. Leave the Cypress source authoritative during dual-run and append the port result to this file's `Port history`; deletion is outside the port task.
