Lesson 0004 · Pivot queries

Lightdash pivot SQL stays long; the backend transform makes it wide

The pivot query builder usually does not create one SQL column per pivot value. It wraps semantic SQL, assigns row and column indexes, and lets streaming backend code widen the result.

Previous: PoP Intro Reference
What you should be able to do after this lesson Explain how PivotConfiguration becomes original_query → group_by_query → pivot_query, why row_index/column_index exist, and how AsyncQueryService.runQueryAndTransformRows widens rows.

1. Two different meanings of “pivot”

Classic SQL pivot means dynamic values become physical SQL columns:

-- conceptual warehouse pivot
status values: completed, pending
columns: revenue_completed, revenue_pending

Lightdash's backend pivot path is different. QueryBuilder/CLAUDE.md states that PivotQueryBuilder does not perform the final pivot. It generates SQL that tags each row with row_index and column_index; the actual spreading happens downstream in AsyncQueryService.runQueryAndTransformRows.

flowchart LR Flat[Flat semantic SQL] --> PivotSQL[PivotQueryBuilder
long rows + row_index + column_index] PivotSQL --> Warehouse[Warehouse returns ordered long rows] Warehouse --> Transform[AsyncQueryService.runQueryAndTransformRows] Transform --> Wide[Wide pivot result rows]

2. The source of truth: PivotConfiguration

PivotConfiguration lives in packages/common/src/types/pivot.ts. It is derived from chart config and metric query in derivePivotConfigFromChart.ts.

4. The normal full-pivot pipeline

PivotQueryBuilder.toSql removes the base query limit, builds group_by_query, then chooses simple or full pivot mode depending on whether groupByColumns exist.

flowchart TD A[Base SQL from MetricQueryBuilder] --> B[original_query] B --> C[group_by_query
group by index + groupBy + hidden helpers
aggregate values] C --> D[pivot_query
select fields + row_index + column_index] D --> E[filtered_rows
row_index <= row limit] E --> F[distinct_groups
distinct pivot header combinations] F --> G[total_columns] D --> H[final SELECT p.*, total_columns
ordered by row_index, column_index] G --> H H --> I[AsyncQueryService transform] 
WITH original_query AS (
  -- MetricQueryBuilder SQL
),

group_by_query AS (
  SELECT
    "month",
    "status",
    ANY_VALUE("revenue") AS "revenue_any"
  FROM original_query
  GROUP BY "month", "status"
),

pivot_query AS (
  SELECT
    g."month" AS "month",
    g."status" AS "status",
    g."revenue_any" AS "revenue_any",
    DENSE_RANK() OVER (ORDER BY g."month" ASC) AS "row_index",
    DENSE_RANK() OVER (ORDER BY g."status" ASC) AS "column_index"
  FROM group_by_query g
),

filtered_rows AS (
  SELECT * FROM pivot_query WHERE "row_index" <= 500
),

distinct_groups AS (
  SELECT DISTINCT "status" FROM filtered_rows
),

total_columns AS (
  SELECT COUNT(*) AS total_columns FROM distinct_groups
)

SELECT p.*, t.total_columns
FROM pivot_query p
CROSS JOIN total_columns t
WHERE p."row_index" <= 500
ORDER BY p."row_index", p."column_index";

5. group_by_query: pivot grain

group_by_query is where rows are collapsed to the pivot grain:

one row per (index columns + pivot header columns + needed hidden helper dims)

For table charts, value columns often use ANY_VALUE because the metric was already semantically aggregated by MetricQueryBuilder. The pivot layer is not redefining revenue; it is regrouping the already-produced result for display.

Boundary reminder If revenue is semantically wrong before pivoting, group_by_query will preserve or aggregate a wrong value. Pivot code is not responsible for fanout-safe metric semantics.

6. row_index and column_index

The final streaming transform needs rows ordered so it can accumulate one output object per rendered row. row_index identifies the rendered row. column_index identifies the pivot header order.

row_index

Dense rank over index-column order. Example: Jan = 1, Feb = 2.

column_index

Dense rank over pivot-column order. Example: completed = 1, pending = 2.

When there are no index columns, all rows can have row_index = 1; when there are no groupByColumns, PivotQueryBuilder takes a simpler path and does not need the full row/column index machinery.

7. Streaming transform: long rows become wide rows

AsyncQueryService.runQueryAndTransformRows receives warehouse rows in row_index, column_index order. It holds a current output row. When row_index changes, it flushes the previous output row and starts a new one.

sequenceDiagram participant DB as Warehouse rows participant S as AsyncQueryService participant Out as Output stream/file DB->>S: row_index=1, status=completed, revenue_any=100 S->>S: start row { month: Jan } S->>S: set revenue_any_completed = 100 DB->>S: row_index=1, status=pending, revenue_any=30 S->>S: set revenue_any_pending = 30 DB->>S: row_index=2, status=completed, revenue_any=120 S->>Out: flush Jan row S->>S: start row { month: Feb }

The pivoted value column name is built from:

value column field name + '_' + joined raw pivot values

orders_revenue_any + '_' + completed
= orders_revenue_any_completed

valuesColumnData records metadata for each generated pivot column, including the original reference field, aggregation, pivot values, and columnIndex.

8. What total_columns means

The SQL returns total_columns so the frontend/backend can know how many pivot columns exist before column limiting. distinct_groups counts distinct pivot header combinations after row filtering. If there are multiple value columns and metrics are not displayed as rows, the count is multiplied by value count.

distinct pivot statuses = completed, pending = 2
valuesColumns = revenue, order_count = 2
metricsAsRows = false

total_columns = 2 * 2 = 4

9. Sorting by dimensions: relatively simple

If sorting only uses index or group-by dimensions, pivot_query can compute DENSE_RANK() inline:

DENSE_RANK() OVER (ORDER BY g."month" ASC) AS "row_index",
DENSE_RANK() OVER (ORDER BY g."status" ASC) AS "column_index"

Custom bin dimensions may carry an _order helper column through the pipeline when actively sorted.

10. Sorting by metrics: the anchor problem

Metric sorting in a pivot is harder because each rendered row has many metric cells. If you sort rows by revenue, which pivot column's revenue should decide the row order?

monthrevenue_completedrevenue_pending
Jan100300
Feb20020

If sorting descending by revenue, Jan wins if pending is the anchor, but Feb wins if completed is the anchor. Lightdash needs a deterministic anchor column.

Anchor column When sorting rows by a value column, Lightdash sorts rows by that metric's value in one selected pivot column. By default this is the first pivot column after column ordering. A pinned sort can specify a particular pivot value.

11. Metric sorting CTE pipeline

PivotQueryBuilder builds extra CTEs for metric sorting:

CTERole
Column anchors *_caCompute first metric values per pivot-column group to support column ordering.
column_rankingAssign col_idx to distinct pivot header combinations.
anchor_columnPick the pivot header used as row-sort anchor, usually col_idx = 1.
Row anchors *_raFor each rendered row, get the metric value at the anchor column.
row_rankingAssign row_index using row anchor values.
flowchart TD GB[group_by_query] --> CA[metric column anchors
*_ca] CA --> CR[column_ranking
DENSE_RANK pivot headers] CR --> AC[anchor_column
pick col_idx=1 or pinned value] GB --> RA[row anchors
*_ra metric value at anchor] AC --> RA RA --> RR[row_ranking
DENSE_RANK rows] CR --> PQ[pivot_query] RR --> PQ

The local QueryBuilder/CLAUDE.md notes that this precomputed ranking path also helps warehouses like Databricks/Spark where inline CTE expansion can make window-function aliases hard to resolve.

12. Three pivot modes to recognize

13. Hidden sort-only and passthrough fields

derivePivotConfigFromChart routes hidden fields carefully:

  • sortOnlyColumns: hidden metrics/table calculations needed for sorting but not display.
  • sortOnlyDimensions: hidden pivot dimensions that affect column order but do not become visible headers.
  • passthroughDimensions: hidden dimensions that do not sort but must flow through for rich text or image templates.

These fields explain some “why is this hidden field in the SQL?” surprises. It may be there to preserve sorting or templates, not to render as a pivot header.

14. Exercises

Does PivotQueryBuilder usually emit revenue_completed as a SQL column?

No. It emits long rows with status = completed, revenue_any, row_index, and column_index. The backend transform creates the wide column name.

What does group_by_query group by?

Visible pivot group-by columns, sort-only dimensions, passthrough dimensions, and index columns. It aggregates values with the configured aggregation.

Why is row_index needed?

It lets the streaming transform know which warehouse rows belong to the same rendered pivot row.

Why is metric sorting ambiguous in a pivot?

A row has multiple metric values across pivot columns. Lightdash must choose an anchor pivot column whose value controls row order.

What file turns long pivot rows into wide rows?

packages/backend/src/services/AsyncQueryService/AsyncQueryService.ts, in runQueryAndTransformRows.
