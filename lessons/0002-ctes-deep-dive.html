Lesson 0002 · CTE composition

CTEs are named scopes: Lightdash query magic is scope management

Generated Lightdash SQL looks scary because it stacks many CTEs. The way through is to track each CTE's input, output columns, and alias scope.

Previous: Fanouts Next: PoP Reference
What you should be able to do after this lesson Trace one output alias through a generated Lightdash query and identify which CTE created it, which CTEs can reference it, and why sibling CTEs cannot see it unless they join or select from it.

1. Plain SQL CTEs: the only rule you need

A CTE is a named subquery. Later CTEs can read earlier CTEs. Sibling CTEs do not share columns by magic. A column alias exists only in the result of the CTE that selected it, and in later queries that read from that CTE.

WITH revenue_by_month AS (
  SELECT month, SUM(amount) AS revenue
  FROM orders
  GROUP BY month
),

large_months AS (
  SELECT month
  FROM revenue_by_month
  WHERE revenue > 1000
)

SELECT * FROM large_months;
flowchart TD A[revenue_by_month
columns: month, revenue] --> B[large_months
reads revenue_by_month
columns: month] B --> C[final SELECT] D[Some sibling CTE] -. cannot see revenue unless it FROM/JOINs A .-> A
Scope rule If you see some_alias used inside a CTE, ask: “Is the CTE selecting from a source that exposes some_alias?” If not, the alias is out of scope.

2. Why Lightdash uses so many CTEs

Lightdash has to take semantic-layer definitions and produce one warehouse query. Each semantic feature wants a different intermediate shape:

  • Fanout protection needs entity keys and safe aggregates.
  • Distinct metrics need row-numbered deduplication by distinct keys.
  • Period-over-period needs current rows and shifted comparison rows.
  • Pivoting needs long rows ranked by row and column order.
  • Table calculations and post-aggregation metrics may need outer selects.

CTEs make those shapes explicit and composable.

4. Fanout CTEs as scopes

From the fanout lesson, a protected table produces cte_keys_customers and cte_metrics_customers. The important scope point: cte_metrics_customers can read columns from cte_keys_customers because it explicitly has FROM cte_keys_customers.

cte_keys_customers AS (
  SELECT DISTINCT
    orders.status AS "orders_status",
    customers.customer_id AS "pk_customer_id"
  FROM customers
  LEFT JOIN orders ON ...
),

cte_metrics_customers AS (
  SELECT
    cte_keys_customers."orders_status",
    SUM(customers.lifetime_value) AS "customers_total_ltv"
  FROM cte_keys_customers
  LEFT JOIN customers
    ON cte_keys_customers."pk_customer_id" = customers.customer_id
  GROUP BY 1
)

But cte_metrics_customers cannot read aliases from cte_unaffected unless it joins cte_unaffected. Sibling CTEs are independent.

5. Distinct metric CTEs as scopes

sum_distinct and average_distinct get dd_* CTEs through buildDistinctMetricCtes. The code comments describe the semantics:

  1. Inner subquery picks one row per distinct_keys combination with ROW_NUMBER().
  2. Outer CTE aggregates up to the grain of selected dimensions that overlap distinct keys.
  3. The result joins back to dd_base by overlapping keys, or cross joins if scalar.
dd_orders_deduped_revenue AS (
  SELECT
    SUM(CASE WHEN __dd_rn = 1 THEN __dd_val ELSE NULL END)
      AS "orders_deduped_revenue"
  FROM (
    SELECT
      orders.amount AS __dd_val,
      ROW_NUMBER() OVER (
        PARTITION BY orders.order_id
        ORDER BY orders.amount
      ) AS __dd_rn
    FROM ...joined query...
  ) __dd_sub
),

dd_base AS (
  -- whatever the query had before distinct metrics were stitched in
),

SELECT
  dd_base.*,
  dd_orders_deduped_revenue."orders_deduped_revenue"
FROM dd_base
CROSS JOIN dd_orders_deduped_revenue
Alias lesson A dd_* metric alias is available only after the query reads from the dd_* CTE. If some earlier sibling CTE emits a derived metric that expects the dd_* alias, the alias is not in scope there.

6. PoP CTEs as scopes

PoP creates a current scope and a previous-period scope. In the non-fanout path, base_metrics freezes current rows. pop_min_max_* reads base_metrics. pop_metrics_* reads the base tables again, constrained by shifted bounds. The final select reads both base_metrics and pop_metrics_*.

flowchart TD Current[normal current SELECT] --> Base[base_metrics] Base --> MinMax[pop_min_max_*] MinMax --> Pop[pop_metrics_*] Base --> Final[final SELECT] Pop --> Final

In the fanout path, the same idea appears as cte_pop_min_max_*, cte_pop_keys_*, and cte_pop_metrics_*. The prefix changes because it must respect the fanout-protected key/metric split.

7. Pivot CTEs as scopes

Pivot CTEs are added after the semantic SQL is built. original_query exposes the flat result from MetricQueryBuilder. group_by_query groups that output to pivot grain and gives value aliases suffixes like _any. pivot_query reads group_by_query and adds ranks.

WITH original_query AS (
  -- MetricQueryBuilder output
),

group_by_query AS (
  SELECT
    "month",
    "status",
    ANY_VALUE("revenue") AS "revenue_any"
  FROM original_query
  GROUP BY "month", "status"
),

pivot_query AS (
  SELECT
    g."month",
    g."status",
    g."revenue_any",
    DENSE_RANK() OVER (ORDER BY g."month") AS "row_index",
    DENSE_RANK() OVER (ORDER BY g."status") AS "column_index"
  FROM group_by_query g
)

8. How to read a generated query top-down

Top-down reading answers: “What intermediate tables are being created?”

  1. Scan the CTE names in order.
  2. Classify each CTE family: fanout, distinct, PoP, pivot, table calc/post-agg.
  3. For each CTE, list its output aliases from the SELECT clause.
  4. Note which later CTEs read it with FROM or JOIN.
Top-down cheat: cte_keys_* → exposes dimension aliases + pk_* cte_metrics_* → exposes dimension aliases + protected metrics dd_* → exposes joinable dimension aliases + one distinct metric dd_base → exposes everything before dd stitching base_metrics → exposes current-period rows pop_metrics_* → exposes previous-period metric aliases original_query → exposes flat semantic result to pivot group_by_query → exposes pivot grain + value aliases pivot_query → exposes pivot grain + row_index + column_index

9. How to read a generated query bottom-up

Bottom-up reading answers: “Where did this final output alias come from?” This is usually better for debugging.

Alias tracing algorithm
  1. Pick one final output alias, e.g. "orders_revenue__pop__month_1".
  2. Search the SQL for AS "orders_revenue__pop__month_1".
  3. If multiple places emit it, you found a duplication risk.
  4. For the intended emission, identify the CTE or final select that owns it.
  5. List every source alias used in its expression.
  6. For each source alias, ask: is it in scope via FROM/JOIN?
  7. Repeat until you reach base table columns or metric compiled SQL.

10. A miniature alias trace

WITH cte_unaffected AS (
  SELECT
    "orders".status AS "orders_status",
    SUM("orders".amount) AS "orders_total_amount"
  FROM ...
  GROUP BY 1
),

cte_metrics_customers AS (
  SELECT
    cte_keys_customers."orders_status",
    SUM("customers".lifetime_value) AS "customers_total_ltv"
  FROM cte_keys_customers
  LEFT JOIN "customers" ON ...
  GROUP BY 1
)

SELECT
  cte_unaffected."orders_status",
  cte_unaffected."orders_total_amount",
  cte_metrics_customers."customers_total_ltv"
FROM cte_unaffected
INNER JOIN cte_metrics_customers
  ON cte_unaffected."orders_status" = cte_metrics_customers."orders_status"

Trace customers_total_ltv:

  1. Final select emits cte_metrics_customers."customers_total_ltv".
  2. cte_metrics_customers created that alias with SUM(customers.lifetime_value).
  3. It was safe because it read from cte_keys_customers and joined back to customers by primary key.

11. Why Lightdash rewrites metric references

Semantic metrics can reference other metrics, e.g. ${orders.profit} / NULLIF(${orders.revenue}, 0). Once metrics live in CTEs, the raw semantic reference must become a CTE-qualified SQL reference. replaceMetricReferencesWithCteReferences handles this in MetricQueryBuilder.

Raw semantic reference

${orders.revenue}

Not valid SQL inside an outer CTE unless compiled/replaced.

CTE-qualified reference

cte_unaffected."orders_revenue"

Valid if the current scope reads cte_unaffected.

12. Practice

Can cte_metrics_customers use an alias created in cte_unaffected?

No, unless cte_metrics_customers explicitly selects from or joins cte_unaffected. They are siblings.

Why does dd_base exist?

It freezes the query result before distinct metrics are stitched in, so dd_* outputs can be joined to a stable base scope.

What does group_by_query read from?

original_query, which is the flat semantic result from MetricQueryBuilder.

If an alias appears as AS "x" in two CTEs, what should you ask?

Which CTE is supposed to own x? If both are later selected, the final result may have duplicate or ambiguous columns.

Why do CTE joins often use dimension aliases?

Fanout/PoP/metric CTEs produce metric values at the result grain. Dimension aliases are the keys for stitching those metric values back together.
