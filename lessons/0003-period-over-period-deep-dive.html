Lesson 0003 · Period-over-period

Period-over-period is a shifted self-join of metric results

A PoP metric is not just a label or a table calculation. Backend PoP reruns the base metric over a shifted time window and joins previous-period rows back to current-period rows.

Previous: CTEs Next: Pivot Reference
What you should be able to do after this lesson Explain why PoP requires a selected comparison time dimension, why it uses min/max CTEs, why date filters are shifted, and why previous-period rows are left-joined back to current rows.

1. User-facing idea

The Lightdash guide describes period-over-period as comparing a metric to a previous time period, like month-over-month or year-over-year (PoP guide). The beta period comparison flow asks the user to select:

  • a time/date dimension,
  • a metric,
  • an offset such as previous month or previous year,
  • then adds a generated metric to the chart.

The backend representation is an AdditionalMetric with PoP metadata, built in periodOverPeriodComparison.ts.

{
  table: 'orders',
  name: 'revenue__pop__month_1__abc123',
  label: 'Revenue (Previous month)',
  type: MetricType.SUM,
  sql: '${TABLE}.amount',
  hidden: true,
  generationType: 'periodOverPeriod',
  baseMetricId: 'orders_revenue',
  timeDimensionId: 'orders_order_date_month',
  granularity: TimeFrames.MONTH,
  periodOffset: 1
}

2. What the metadata means

FieldMeaningExample

periodOffsetHow many granularity units backward.1 for previous month, 12 for same month last year if granularity is month.
Compile-time guard MetricQueryBuilder requires the PoP comparison time dimension to be selected in the query. Without it, the final join cannot align current and previous rows.

4. Non-fanout PoP SQL shape

When fanout protection does not take over the metric path, MetricQueryBuilder wraps the normal current query as base_metrics, computes current min/max dates, reruns the base metric over shifted bounds, and left-joins the previous rows back.

flowchart TD A[Current metric SELECT] --> B[base_metrics] B --> C[pop_min_max_month_1
MIN/MAX base month] C --> D[pop_metrics_month_1
same metric SQL, shifted date filter] B --> E[final SELECT] D --> E E --> F[current + previous metric columns] 
WITH base_metrics AS (
  SELECT
    DATE_TRUNC('MONTH', "orders".order_date) AS "orders_order_date_month",
    SUM("orders".amount) AS "orders_revenue"
  FROM "orders" AS "orders"
  WHERE DATE_TRUNC('MONTH', "orders".order_date)
    BETWEEN '2026-03-01' AND '2026-05-01'
  GROUP BY 1
),

pop_min_max_month_1 AS (
  SELECT
    MIN(base_metrics."orders_order_date_month") AS min_date,
    MAX(base_metrics."orders_order_date_month") AS max_date
  FROM base_metrics
),

pop_metrics_month_1 AS (
  SELECT
    DATE_TRUNC('MONTH', "orders".order_date) AS "orders_order_date_month",
    SUM("orders".amount) AS "orders_revenue__pop__month_1"
  FROM "orders" AS "orders"
  LEFT JOIN pop_min_max_month_1 ON TRUE
  WHERE DATE_TRUNC('MONTH', "orders".order_date) >= pop_min_max_month_1.min_date - INTERVAL '1 MONTH'
    AND DATE_TRUNC('MONTH', "orders".order_date) <= pop_min_max_month_1.max_date - INTERVAL '1 MONTH'
  GROUP BY 1
)

SELECT
  base_metrics.*,
  pop_metrics_month_1."orders_revenue__pop__month_1"
FROM base_metrics
LEFT JOIN pop_metrics_month_1
  ON base_metrics."orders_order_date_month"
   = pop_metrics_month_1."orders_order_date_month" + INTERVAL '1 MONTH';

5. Why the min/max CTE exists

PoP compares the currently visible time window to a shifted window. Instead of hard-coding the request filter, Lightdash derives the current time bounds from base_metrics:

SELECT MIN(base.month), MAX(base.month) FROM base_metrics

Then it computes the comparison window by shifting those bounds. If current months are March–May and offset is one month, previous months are February–April.

timeline title Month-over-month windows 2026-02 : previous window begins 2026-03 : current window begins; previous rows join to March/April/May 2026-04 : overlap month 2026-05 : current window ends

6. Filter semantics: reuse non-time filters, shift time filters

The previous-period query should answer the same business question in a different time window. So non-time filters are reused; the comparison time dimension's current date condition is replaced by shifted bounds.

FilterCurrent queryPrevious-period query
orders_status = 'completed'KeepKeep
country = 'US'KeepKeep
month between Mar and MayKeepReplace with shifted Feb to Apr bounds

In code, this appears as getPopDimensionsFilterSQL(popFieldId) plus interval syntax built from cfg.periodOffset and cfg.granularity.

7. The final join condition

For non-time dimensions, current and previous rows match by null-safe equality. For the comparison time dimension, the previous row is shifted forward to equal the current row.

LEFT JOIN pop_metrics_month_1 ON
  base_metrics."orders_status" = pop_metrics_month_1."orders_status"
  AND base_metrics."orders_order_date_month"
      = pop_metrics_month_1."orders_order_date_month" + INTERVAL '1 MONTH'
PoP invariant The previous-period CTE should produce the same grouping dimensions as the current rows. All dimensions join by equality, except the comparison time dimension joins by interval-shifted equality.

8. Why LEFT JOIN?

A current row should not disappear just because the previous period has no matching row. If May has revenue but April has none, the output should show May with NULL previous revenue. That requires LEFT JOIN.

Inner join behavior

Current rows without previous rows vanish. That hides real current-period data.

Left join behavior

Current rows remain. Previous metric is null when the shifted row is missing.

9. Multiple PoP configs

MetricQueryBuilder groups selected/filtered PoP metrics by comparison config: timeDimensionId, granularity, and periodOffset. The config gets a short hashed CTE suffix so multiple comparisons can exist in one query, e.g. previous month and same month last year.

getPopComparisonConfigKey({
  timeDimensionId: 'orders_order_date_month',
  granularity: TimeFrames.MONTH,
  periodOffset: 1
})

cte suffix: month_1__00abc123

Metrics with the same config can share min/max and shifted-date CTEs; metrics with different configs need separate shifted windows.

10. Fanout-aware PoP: same shifted idea, safer metric path

If the base metric is in a table that needs fanout protection, Lightdash cannot simply rerun a flat previous-period aggregate. It must use the same protected-key pattern in the shifted period.

flowchart TD K[cte_keys_table
current distinct dimensions + pk] --> M[cte_metrics_table
current protected metric] K --> MM[cte_pop_min_max_table
current min/max comparison time] MM --> PK[cte_pop_keys_table
previous-period distinct dimensions + pk] PK --> PM[cte_pop_metrics_table
previous protected metric] M --> Final[final fanout SELECT] PM --> Final Final --> Out[current metric + previous protected metric]

The crucial difference is that cte_pop_keys_* is not just a copy of cte_keys_*. It selects keys from the shifted previous-period window, because the relevant primary keys may differ between current and previous periods.

12. Why not just use LAG()?

LAG() is row-position based. It compares to “the previous row in the visible ordered result”. Backend PoP is time-window based. It computes the metric over actual shifted dates, with the same non-time filters and grouping dimensions. That is more robust when rows are missing, filters change, or the offset is not simply one visible row.

LAG()

Depends on result row order and presence of rows. Good for some table calculations.

Backend PoP

Reruns metric over shifted time bounds and joins by interval. Better semantic comparison.

13. Exercises

Current window: March–May. Granularity: month. Offset: 1. What previous window is queried?

February–April. Those previous rows join forward to March–May.

Current month row is May 2026. Granularity month, offset 12. Which previous row joins?

May 2025, because May 2025 + 12 months = May 2026.

Why must orders_order_date_month be selected for a monthly PoP metric?

The final join needs the current row's month to match it to the shifted previous row.

Why do previous-period CTEs reuse country/status filters?

The comparison should answer the same business question in a different time window, not a different segment.

Why does fanout-aware PoP create cte_pop_keys_*?

Previous-period primary keys can differ from current-period primary keys, and protected metrics must be aggregated at a safe key grain in the shifted window.
