Lesson 0001 · Fanouts

Fanouts are aggregate bugs caused by changing row grain

A fanout is not mysterious. It is a join that makes rows from one table appear multiple times. If you aggregate those repeated rows, some metrics lie.

Intro Next: CTEs Reference
What you should be able to do after this lesson Given a base table, a join relationship, and a metric, predict whether the metric can inflate and explain why Lightdash may build cte_keys_* and cte_metrics_*.

1. Grain: the unit of one row

Grain means what one row represents. A table can have customer grain, order grain, line-item grain, account grain, session grain, and so on. A join changes the result grain when one row on one side matches multiple rows on the other side.

If the Explore starts from customers and left-joins orders, customer 1 appears twice:

SELECT
  customers.customer_id,
  customers.lifetime_value,
  orders.order_id,
  orders.amount
FROM customers
LEFT JOIN orders ON customers.customer_id = orders.customer_id;

-- joined result
-- customer_id | lifetime_value | order_id | amount
-- 1           | 100            | 10       | 20
-- 1           | 100            | 11       | 30
-- 2           | 70             | 12       | 15

The joined result is no longer one row per customer. It is closer to order grain, with customer fields repeated.

2. Metric inflation: repeated rows enter the aggregate

Now compare two metrics:

Total order amountSUM(orders.amount)20 + 30 + 15 = 65Yes
Total customer LTVSUM(customers.lifetime_value)100 + 100 + 70 = 270No; true total is 170

The customer metric inflates because the join duplicated customer rows. The order metric did not inflate because the join result still has one row per order.

3. Relationship names are directional

The Lightdash joins docs emphasize direction: “Accounts joining to Users (one-to-many) is completely different from Users joining to Accounts (many-to-one).” The relationship is from the current/starting table toward the joined table.

RelationshipQuestion to askFanout intuition
one-to-manyFor one starting row, can I find many joined rows?Starting-table fields can repeat.
many-to-oneFor many starting rows, do they each find one joined row?Joined-table fields can repeat across starting rows; starting rows usually do not multiply.
one-to-oneDoes each side match at most one row?No multiplication expected.
many-to-manyCan multiple rows on each side match multiple rows?Assume broad fanout risk.

Lightdash docs recommend defining relationship and primary_key in YAML so Lightdash can understand the intended row-level granularity and prevent metric inflation (Handling fanouts).

type: model
name: customers
primary_key: customer_id

joins:
  - join: orders
    sql_on: ${customers.customer_id} = ${orders.customer_id}
    relationship: one-to-many

4. Which metric types are vulnerable?

In QueryBuilder/utils.ts, Lightdash marks these as inflation-proof: COUNT_DISTINCT, SUM_DISTINCT, AVERAGE_DISTINCT, MIN, and MAX. That does not mean every query is magically easy; it means repeated rows do not affect those metrics the same way plain SUM/AVG are affected.

Metric typeRepeated rows effectExample
SUMUsually unsafe if summing repeated one-side rows.SUM(customers.lifetime_value) becomes 270.
AVGCan be unsafe because repeated entities get extra weight.Customer 1's LTV is counted twice in the average.
COUNTCounts joined rows, so may or may not match intent.COUNT(customers.customer_id) after order join counts order rows with customer IDs.
COUNT_DISTINCTRepeated IDs collapse.COUNT_DISTINCT(customers.customer_id) remains 2.
MIN/MAXRepeating a value does not change min/max.MAX(customers.lifetime_value) remains 100.
SUM_DISTINCT/AVERAGE_DISTINCTHandled by explicit distinct-key deduplication.Uses dd_* CTEs, covered in the CTE lesson.
Important nuance “Inflation-proof” is about repeated identical source rows under join fanout. It does not prove the metric matches business intent. For example, COUNT_DISTINCT(order_id) is mathematically stable under repeated joins, but it may still answer the wrong question if the distinct key is wrong.

5. How Lightdash detects tables with possible metric inflation

MetricQueryBuilder.getExperimentalMetricsCteSQL calls findTablesWithMetricInflation with tables, joins, base table, and joined tables. That function uses join relationships to identify tables whose metrics may inflate in the current query. If a join lacks a relationship, Lightdash warns because it cannot safely infer the fanout risk.

flowchart TD A[Selected dimensions/metrics/filters] --> B[Which tables are required?] B --> C[Which joins are needed?] C --> D[findTablesWithMetricInflation] D --> E{Table has primary key?} E -- no --> W[Warn: cannot deduplicate safely] E -- yes --> F[Build fanout protection CTEs]

6. The Lightdash fanout protection pattern

For each table whose metrics can inflate, Lightdash builds two CTEs:

  1. cte_keys_<table>: from the full joined query, select distinct output dimensions plus the protected table's primary key.
  2. cte_metrics_<table>: join those keys back to only the protected table by primary key and aggregate the table's vulnerable metrics.
flowchart TD J[Full joined world
dimensions, filters, all joins] --> K[cte_keys_customers
DISTINCT dimensions + customers.pk] K --> M[cte_metrics_customers
join keys back to customers by pk] M --> Safe[Safe customer SUM/AVG/etc] J --> U[cte_unaffected
dimensions + safe or unaffected metrics] Safe --> Final[final SELECT] U --> Final 
WITH cte_keys_customers AS (
  SELECT DISTINCT
    "orders".status AS "orders_status",
    "customers".customer_id AS "pk_customer_id"
  FROM "customers" AS "customers"
  LEFT JOIN "orders" AS "orders"
    ON "customers".customer_id = "orders".customer_id
),

cte_metrics_customers AS (
  SELECT
    cte_keys_customers."orders_status",
    SUM("customers".lifetime_value) AS "customers_total_ltv"
  FROM cte_keys_customers
  LEFT JOIN "customers" AS "customers"
    ON cte_keys_customers."pk_customer_id" = "customers".customer_id
  GROUP BY 1
),

cte_unaffected AS (
  SELECT
    "orders".status AS "orders_status",
    SUM("orders".amount) AS "orders_total_amount"
  FROM "customers" AS "customers"
  LEFT JOIN "orders" AS "orders"
    ON "customers".customer_id = "orders".customer_id
  GROUP BY 1
)

SELECT
  cte_unaffected.*,
  cte_metrics_customers."customers_total_ltv"
FROM cte_unaffected
INNER JOIN cte_metrics_customers
  ON cte_unaffected."orders_status" = cte_metrics_customers."orders_status";

7. Why not just SUM(DISTINCT lifetime_value)?

Because distinct values are not distinct entities. If two different customers both have LTV 100, SUM(DISTINCT lifetime_value) counts 100 once, not twice. Fanout protection must deduplicate by entity key, not by metric value.

Wrong dedup key: value

SUM(DISTINCT lifetime_value)
-- customer A = 100
-- customer B = 100
-- result: 100 (wrong)

Right dedup key: primary key

SELECT DISTINCT customer_id
-- A, B
SUM(lifetime_value)
-- result: 200

8. How dimensions affect the safe grain

If you group by an order dimension like orders.status, cte_keys_customers may contain the same customer primary key under multiple statuses. That is not necessarily wrong: the output question is now “customer LTV by order status”, and the metric is repeated across statuses because each status group has relevant customer keys. Fanout protection prevents duplicate keys within a group; it does not decide whether the analysis itself makes business sense.

Mental model Fanout protection deduplicates protected table primary keys within each output dimension group. It does not globally force a customer to appear in only one result row if your selected dimensions intentionally split the result into multiple groups.

9. Prediction exercises

Base: customers. Join: customers → orders one-to-many. Metric: SUM(customers.lifetime_value). Inflates?

Yes. Customer rows are repeated by orders; plain sum sees the repeated customer values.

Base: customers. Join: customers → orders one-to-many. Metric: SUM(orders.amount). Inflates?

Usually no from this join alone. The joined result is at order grain for orders, so each order amount appears once.

Base: orders. Join: orders → customers many-to-one. Metric: SUM(orders.amount). Inflates?

No from the many-to-one join. Each order finds one customer; order rows are not multiplied.

Base: orders. Join: orders → order_items one-to-many. Metric: AVG(orders.total_amount). Inflates?

Yes. Orders with more items appear more times and get extra weight in the average.

Base: orders. Join: orders → order_items one-to-many. Metric: COUNT_DISTINCT(orders.order_id). Inflates?

No. Repeated order IDs collapse under COUNT_DISTINCT.

Why does Lightdash need primary_key for fanout protection?

Because cte_keys_* must select a distinct entity key and then rejoin to the protected table by that key. Without a primary key, Lightdash cannot know which rows represent one entity.
