Lesson 0000 · System map

Lightdash query generation is a pipeline, not one query builder

The backend SQL that reaches a warehouse is assembled by several systems. Each system has one job: semantic metrics, fanout protection, distinct deduplication, period comparison, pivot indexing, or result transformation.

1. The big architecture map

The local query-builder guide says there are three main builders: MetricQueryBuilder, PivotQueryBuilder, and SqlQueryBuilder. For Explore/Chart metric queries, the important path is usually MetricQueryBuilder → optional PivotQueryBuilder → AsyncQueryService.

2. Start with a concrete chart

Imagine a user asks for a pivot table:

  • Rows: orders_order_date_month
  • Pivot columns: orders_status
  • Metric: orders_revenue
  • Period comparison: previous month revenue
  • Explore has joins that may fan out some metrics

3. Minimum vocabulary

Metric query

The request shape: explore name, dimensions, metrics, filters, sorts, limit, table calculations, optional additional metrics, and optional pivot dimensions.

Compiled metric query

The query after custom metrics, custom dimensions, and table calculations are compiled. PoP generated metrics appear in additionalMetrics and compiled metric maps.

Explore

The semantic model: base table, joined tables, dimensions, metrics, primary keys, and join relationships.

Grain

What one row represents. Fanout bugs are usually grain mismatch bugs.

Fanout

A join that multiplies rows from some table. Multiplication can inflate SUM and AVG.

CTE

A named SQL subquery in a WITH clause. In Lightdash, CTEs are the boundaries between query-builder systems.

PoP

Period-over-period: a generated metric that reruns a base metric over a shifted time window and joins it back to current rows.

Pivot long rows

Rows that still have a pivot header value column plus row_index and column_index. They are not yet final wide pivot rows.

4. The simplest non-pivot query path

If there are no fanouts, no distinct metrics, no PoP, and no pivoting, MetricQueryBuilder can produce familiar SQL:

SELECT
  DATE_TRUNC('MONTH', "orders".order_date) AS "orders_order_date_month",
  SUM("orders".amount) AS "orders_revenue"
FROM "analytics"."orders" AS "orders"
WHERE "orders".is_completed = true
GROUP BY 1
ORDER BY "orders_order_date_month" ASC
LIMIT 500

Most complexity is conditional. Lightdash adds layers only when the query asks for semantics that cannot be expressed safely in that simple shape.

7. Recall checkpoints

Which class owns semantic metric SQL?

MetricQueryBuilder. It builds dimensions, metric aggregates, joins, filters, and semantic CTEs like fanout, distinct, and PoP CTEs.

Which class adds row_index and column_index?

PivotQueryBuilder, specifically through pivot_query and related ranking CTEs.

Which function widens long pivot rows into output rows?

AsyncQueryService.runQueryAndTransformRows. It groups streamed rows by row_index and writes pivot-value-suffixed columns.

What is the best first question when SQL looks scary?

“Which subsystem owns this CTE or alias?” Then inspect the nearby scope and only then move outward.
